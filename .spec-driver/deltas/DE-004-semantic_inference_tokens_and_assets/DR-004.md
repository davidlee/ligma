# DR-004 вҖ“ Semantic inference, tokens, and assets

## 1. Executive Summary
- **Delta**: [DE-004](./DE-004.md)
- **Status**: draft
- **Last Updated**: 2026-03-11
- **Synopsis**: Populate every null placeholder DE-003 left behind вҖ” role inference, component/variable/asset extraction, semantic flags, text kind вҖ” via a three-layer architecture that preserves extractor independence while adding a post-processing inference stage.

## 2. Problem & Constraints

- **Current Behaviour**: DE-003 normalizes raw Figma JSON into a compact, typed `NormalizedNode` tree. Fields for role, component metadata, variable bindings, asset info, and semantic flags are present but stubbed as `null`/`false`/`"unknown"`. The normalized output is structurally complete but semantically inert вҖ” it can't answer "what is this?" (role), "is this reusable?" (component), "is this token-bound?" (variables), or "should this be exported?" (asset).

- **Drivers / Inputs**:
  - PROD-001.FR-007 (role inference)
  - PROD-001.FR-010 (variable/token extraction)
  - PROD-001.FR-016 (used-token summary)
  - docs/brief.md В§8 (8 normalization heuristic rules)
  - DR-003 В§4 (heuristic rules catalogue, role vocabulary, absence semantics)

- **Constraints / Guardrails**:
  - DE-003 design principles (design-principles.md) remain in force вҖ” especially extractor independence, absence semantics, warnings-not-exceptions, and passthrough isolation
  - `NormalizedNode` schema is fixed (DEC-007) вҖ” DE-004 populates nulls, does not restructure
  - NF-001 size ceiling: `JSON.stringify` length вүӨ 2.0x raw input on representative fixtures
  - NF-003: every heuristic must be an isolated pure function with unit tests
  - ADR-001: zero lint warnings
  - ADR-002: `mise run` quality gate

- **Out of Scope**:
  - Core type classification, layout, text, appearance normalization (DE-003)
  - Outline/context generation (DE-005)
  - Selective expansion (DE-006)
  - Full file-level variable inventory (requires Variables API, Enterprise access)
  - Full file-level style inventory
  - Component property *definitions* on components/component-sets (future enhancement)

## 3. Architecture Intent

### Target Outcomes
1. Every `NormalizedNode` carries a semantically meaningful role (or explicit null = "no match") вҖ” FR-007
2. Component/instance nodes carry metadata sufficient for reusability analysis вҖ” FR-005
3. Variable-bound nodes carry per-node binding information distinguishing literals from tokens вҖ” FR-010
4. A `tokens-used.json` artifact aggregates all encountered token references scoped to the normalized subtree вҖ” FR-016
5. Asset-worthy nodes carry classification and export suggestions вҖ” FR-005
6. Text nodes carry inferred semantic kind (heading/label/body/caption/button) вҖ” FR-009
7. All semantic flags populated from role/type/component derivation вҖ” FR-007
8. Diagnostics include confidence bands and provenance for all inferences вҖ” FR-007

### Architectural Boundaries

**Extraction** (Layer 1): Raw Figma node вҶ’ typed normalized fragment. Each extractor is pure, independent, reads only raw input via `getRawProperty()`. No extractor imports another. This layer is *descriptive*.

**Inference** (Layer 3): Assembled normalized fragments вҶ’ higher-order semantic fields. Operates on a narrow `InferenceInput` type (not the full `NormalizedNode`). This layer is *interpretive*.

**Summary** (post-normalization): Aggregates across the full normalized tree. Lives outside `normalize/`. Consumes normalized output, not raw input.

**Design rule**: Anything that reads raw Figma input belongs in `normalize/`. Anything that aggregates across nodes belongs in `summary/`. The `infer/` subdirectory reads assembled fragments, not raw input. (DE-003 design-principles.md В§2 anticipated this: "DE-004 may add a post-processing stage that combines results across extractors.")

### Inference Execution Order (DEC-031)

Inference is a **top-down post-pass** over the completed extraction tree, not interleaved with assembly. The pipeline is:

1. `normalizeNode(raw, context)` recursively builds the full tree with extraction + assembly only. Role, semantics, and textKind are set to defaults (null/false/unknown).
2. `applyInferencesRecursive(node, parentRole?)` walks the completed tree **top-down**. At each node: infer role, then infer children (passing parent's role as context). This makes `parentRole` available for text-kind inference (e.g., text inside a button-role frame).

Top-down is required because:
- Button/icon detection inspects children вҶ’ children must be assembled (satisfied by step 1)
- Text kind accepts `parentRole` вҶ’ parent must be inferred before children
- Label detects input-like siblings вҶ’ see В§5.6 label rule clarification

The tree is mutated in-place during the inference pass (or rebuilt via shallow copy). Either is acceptable; in-place is simpler since the tree is freshly constructed and not shared.

### Guiding Principles (extending DE-003)
- Extraction is descriptive; inference is interpretive. Different concerns, different layers.
- Inference functions declare their input surface explicitly via `InferenceInput` вҖ” no quiet dependency on fields they should not read.
- `variables` excluded from `InferenceInput` вҖ” inference reads structure and presentation, not binding/summary outputs. Read/write boundary stays crisp.
- Name matching is reinforcing, never sufficient alone.
- Noise is annotation (diagnostic warning), not deletion. Structural truth preserved.

## 5. Detailed Design

### 5.1 Shared Result Types (DEC-021)

```ts
// src/normalize/raw-helpers.ts

type AnalysisResult<T> = {
  value: T
  confidence: Confidence    // 'high' | 'medium' | 'low'
  warnings: string[]
}

type ExtractorResult<T> = AnalysisResult<T> & {
  omittedFields: string[]
}
```

`AnalysisResult<T>` is the universal result contract for both extraction and inference. `ExtractorResult<T>` extends it with `omittedFields` вҖ” an extraction-specific concept (raw fields the extractor chose not to map). Inference functions use the base type.

DE-003 extractors gain explicit `confidence` field. Same logic as current `node.ts` derivation (`warnings.length > 0 ? 'medium' : 'high'`), moved to source. `classify()` returns `AnalysisResult<NormalizedNodeType>`, not `ExtractorResult` вҖ” classification has no omitted fields.

`node.ts` aggregation: takes min of all confidences, concatenates all warnings, concatenates all omittedFields.

### 5.2 `InferenceInput` Type (DEC-022)

```ts
// src/normalize/infer/types.ts

type InferenceInput = Readonly<{
  type: NormalizedNodeType
  name: string
  visible: boolean
  bounds: Bounds | null
  layout: NormalizedLayout | null
  appearance: NormalizedAppearance | null
  text: NormalizedText | null
  component: NormalizedComponentInfo | null
  hierarchy: Readonly<{
    parentId: string | null
    depth: number
    childCount: number
    path: readonly HierarchyEntry[]
  }>
  children: readonly InferenceInput[]
}>

type InferenceResults = {
  role: AnalysisResult<NormalizedRole | null>
  semantics: AnalysisResult<Semantics>
  textKind: AnalysisResult<TextSemanticKind> | null  // null when type !== 'text'
}
```

`InferenceInput` is a named recursive type вҖ” not `Pick<NormalizedNode, ...>`. It documents that inference consumes a semantic view of the assembled node. `variables` excluded: inference reads structure and presentation, not binding outputs. The excluded fields (`role`, `variables`, `asset`, `semantics`, `diagnostics`) are exactly what inference populates.

**Construction**: A `toInferenceInput(node: NormalizedNode): InferenceInput` mapping function strips write-target fields and recursively maps children. Lives in `src/normalize/infer/types.ts` alongside the type definition. Trivial but explicit вҖ” it enforces the boundary at runtime, not just at the type level.

### 5.3 `extractComponent(raw)` (DEC-023)

Returns `ExtractorResult<NormalizedComponentInfo | null>`.

**Returns null** when raw type is not COMPONENT, INSTANCE, or COMPONENT_SET.

**Instances** (`type === 'INSTANCE'`):
- `kind: 'instance'`
- `componentId`: from raw `componentId` (nullable вҖ” medium confidence + warning if missing)
- `componentName`: from raw `name`
- `componentSetId`: from raw `componentSetId` (nullable)
- `propertyValues`: from raw `componentProperties` вҖ” current property values on this instance
- `propertyReferences`: from raw `componentPropertyReferences` вҖ” nested node вҶ’ component property bindings (distinct from instance overrides)
- `isReusable: true`

**Components** (`type === 'COMPONENT'`):
- `kind: 'component'`
- `componentId: null`
- `componentName`: from raw `name`
- `componentSetId`: from raw `componentSetId` (present on variant members)
- `propertyValues: {}` вҖ” no "current values" concept for definitions
- `propertyReferences: {}` вҖ” unless nested refs exist
- `isReusable: true`

**Component sets** (`type === 'COMPONENT_SET'`):
- `kind: 'component-set'`
- `componentId: null`
- `componentName`: from raw `name`
- `componentSetId: null`
- `propertyValues: {}`
- `propertyReferences: {}`
- `isReusable: true`

Component property *definitions* on components/component-sets are deferred. The shape is conceptually different from instance property values and would require a schema extension.

### 5.4 `extractVariables(raw)` (DEC-024)

Returns `ExtractorResult<NormalizedVariableBindings | null>`.

**Returns null** when raw node has no `boundVariables` property.

**Mapping**: Each entry in raw `boundVariables` becomes a binding:

```ts
{
  field: "fills/0/color",        // path to bound property
  tokenId: "VariableID:xxx",     // raw variable ID
  tokenName: null,               // requires Variables API вҖ” null in v1
  collectionId: null,            // extractable from ID format if present, else null
  resolvedType: "color",         // inferred from field path
}
```

**`resolvedType` inference** from field path:
- fills/strokes color fields вҶ’ `'color'`
- padding/gap/spacing/size fields вҶ’ `'number'`
- text content вҶ’ `'string'`
- visibility вҶ’ `'boolean'`
- otherwise вҶ’ `'unknown'`

**`explicitModes`**: Node-level `Record<collectionId, modeId>` from raw `explicitVariableModes`. Stays at the node level вҖ” not attached per-binding, because without reliable collection resolution per variable, per-binding `modeId` would be false precision. Consumers can cross-reference `tokenId` вҶ’ collection вҶ’ `explicitModes` when they have Variables API context.

**Confidence**: `high` when bindings parse cleanly. `medium` when `resolvedType` falls back to `'unknown'`.

### 5.5 `extractAsset(raw)` (DEC-025)

Returns `ExtractorResult<NormalizedAssetInfo | null>`.

**Returns null** when node has no asset signals (no image fills, not a complex vector, no export-suggestive indicators).

**Asset signals** (any triggers extraction):
1. **Image fill present** вҶ’ `kind: 'bitmap'`, `imageRefs` from fill's `imageRef`, confidence `high`
2. **Vector complexity** (heuristic): type is `vector` or `boolean-operation` with child count вүҘ 3 or nested boolean ops вҶ’ `kind: 'svg'`, confidence `medium`, `reason: "vector complexity threshold exceeded вҖ” likely export candidate"`
3. **Naming patterns**: name matches export-suggestive patterns (`icon/*`, `illustration/*`, `img-*`, `asset-*`, etc.) вҖ” reinforcing signal only, never standalone trigger

**Kind resolution**: `'bitmap'` if image fills present; `'svg'` if vector/boolean-op without image fills; `'mixed'` if both. Confidence takes the min.

**`exportSuggested`**: `true` when image fill present OR vector complexity exceeds threshold. `reason` explains why.

Name-pattern-only matches do not trigger asset extraction.

### 5.6 Role Inference вҖ” Priority Rule Chain (DEC-026)

`inferRole(input: InferenceInput)` вҶ’ `AnalysisResult<NormalizedRole | null>`

Runs a prioritized rule chain. First match wins. Each rule is a standalone pure function: `(input: InferenceInput) => AnalysisResult<NormalizedRole> | null` вҖ” returns null if the rule doesn't apply.

**Noise early-out**: Before the rule chain, check for noise conditions (invisible + not structurally important, zero-size bounds, decorative micro-layers). Noise nodes return `null` role with diagnostic warning and appropriate confidence:
- High-confidence null for clearly ignorable nodes (invisible, zero-size): "confident this node has no meaningful role"
- Low-confidence null for ambiguous unmatched nodes at the end of the chain: "uncertain, could not determine"

**Priority order**:

| # | Rule | Type guard | Key signals | Result | Confidence |
| --- | --- | --- | --- | --- | --- |
| 1 | Image | `type === 'image'` | вҖ” | `'image'` | high |
| 2 | Icon | vector/group/bool-op/instance | small bounds (вүӨ48Г—48), square-ish (aspect 0.5вҖ“2.0), no text children | `'icon'` | high structural; medium with name reinforcement |
| 3 | Button | instance/component/frame | single short text child (вүӨ40 chars) OR text+icon, padded auto-layout, visible fill, button-suggestive name | `'button'` / `'icon-button'` | high вүҘ4 signals; medium 2вҖ“3 |
| 4 | Input | frame/instance/component | rectangular (w>h, aspect вүҘ2.0), border/fill, input-suggestive name | `'input'` | high name+structure; medium structure only |
| 5 | Heading | `type === 'text'` | fontSize вүҘ20, fontWeight вүҘ600, short content (вүӨ80 chars) | `'heading'` | high all three; medium fontSize alone |
| 6 | Label | `type === 'text'` | short content (вүӨ40 chars), sibling with input-like structural signals | `'label'` | medium |
| 7 | Body text | `type === 'text'` | longer content (>80 chars) or normal weight + medium size | `'body-text'` | medium |
| 8 | Divider | `type === 'line'` or frame with one dimension вүӨ2px | вҖ” | `'divider'` | high |
| 9 | Grid | вҖ” | `layout.mode === 'grid'` or auto-layout + `wrap: true` | `'grid'` | high |
| 10 | Stack | вҖ” | auto-layout (h/v), children > 0, gap/padding | `'stack'` | high |
| 11 | Card | frame/instance/component | visible fill/stroke, cornerRadius, вүҘ2 mixed-type children | `'card'` | medium |
| 12 | Container | frame/group/component/instance | children > 0, layout/padding | `'container'` | high |
| 13 | Fallback | вҖ” | no rule matched | `null` | low |

**Constraints**:
- Rules 5вҖ“7 (heading/label/body-text) apply only to `type === 'text'` nodes.
- Rule 6 (label): checks sibling *structural signals* (input-suggestive name, rectangular bounds with w>h, border/fill presence) directly from `InferenceInput`, not sibling inferred roles. This keeps the rule self-contained and avoids inference ordering dependencies. Positional co-occurrence, not spatial proximity.
- Icon: name matching is reinforcing only, never sufficient alone. Hard rule.
- Child inspection is one level deep only. No recursive child scanning.
- Name matching is case-insensitive (`input.name.toLowerCase()`).

**Thresholds** (named constants, not magic numbers):

const INPUT_MIN_ASPECT_RATIO = 2.0
const HEADING_MIN_FONT_SIZE = 20
const HEADING_MIN_FONT_WEIGHT = 600
const HEADING_MAX_LENGTH = 80

### 5.7 Text Kind Inference (DEC-027)

`inferTextKind(input: InferenceInput, role: NormalizedRole | null, parentRole?: NormalizedRole | null)` вҶ’ `AnalysisResult<TextSemanticKind>`

Only called when `input.type === 'text'`. Returns `null` otherwise (explicit in function signature and `InferenceResults` type).

**Primary path вҖ” derive from role**:

| Role | вҶ’ TextSemanticKind |
| --- | --- |
| `'heading'` | `'heading'` |
| `'body-text'` | `'body'` |
| `'label'` | `'label'` |
| `'button'` / `'icon-button'` | `'button'` |

Confidence inherits from role inference.

**Parent context path**: If role is null but `parentRole` is `'button'` or `'icon-button'`, text kind is `'button'`. Covers text nodes nested inside button-role frames.

**Fallback heuristic** (when role is null and no parent context):

| Kind | Signals | Confidence |
| --- | --- | --- |
| `'heading'` | fontSize вүҘ 20, fontWeight вүҘ 600, short content | high/medium |
| `'caption'` | fontSize вүӨ 12, or secondary/muted appearance | medium |
| `'body'` | longer text, normal weight/size | medium |
| `'unknown'` | no signals match | medium |

### 5.8 Semantics Inference (DEC-028)

`inferSemantics(input: InferenceInput, role: NormalizedRole | null, component: NormalizedComponentInfo | null)` вҶ’ `AnalysisResult<Semantics>`

Derives 6 boolean flags from role, type, and component. No independent heuristics вҖ” flags are cheap consequences of prior decisions.

| Flag | Derived from |
| --- | --- |
| `likelyInteractive` | role is `'button'`, `'icon-button'`, `'input'`, or `'navigation'` |
| `likelyTextInput` | role is `'input'` |
| `likelyIcon` | role is `'icon'` |
| `likelyImage` | role is `'image'` or `type === 'image'` |
| `likelyMask` | `type === 'mask'` |
| `likelyReusableComponent` | `component !== null && component.isReusable` |

Confidence inherits from role inference confidence (since flags are direct consequences).

### 5.9 Confidence Model (DEC-029)

**Bands**:
- **high**: strong positive match or strong conservative determination, no material competing signals
- **medium**: partial match, missing supporting signals, or extractor degradation that does not undermine the whole node
- **low**: conflicting signals, fallback after failed discrimination, or major unsupported shape that makes interpretation unreliable

**Composition**: `node.ts` takes the minimum confidence across all extraction and inference results. A node with clean extraction but ambiguous role вҶ’ `low`. A node with extractor warning and confident role вҶ’ `medium`. A node with clean extraction and confident role вҶ’ `high`.

**`unknown` is not automatically low**: Sometimes `"unknown"` is the correct, conservative answer with medium confidence. `low` is reserved for genuinely ambiguous or weakly-supported inference.

**Noise confidence**:
- High-confidence null: clearly ignorable (invisible, zero-size) вҖ” "confident this node has no meaningful role"
- Low-confidence null: ambiguous, no rule matched with any support вҖ” "uncertain, could not determine"

### 5.10 Tokens-Used Summary (DEC-030)

`aggregateTokensUsed(root: NormalizedNode, fileKey: string, rootNodeId: string)` вҶ’ `TokensUsedSummary`

Lives in `src/summary/tokens-used.ts`. Called from `orchestrate.ts` after `normalize()` returns.

**Algorithm**:
1. Recursive tree walk over normalized tree
2. For each node with `variables !== null`, iterate `variables.bindings`
3. Deduplicate by `tokenId`:
   - First encounter wins for metadata
   - On `resolvedType` conflict: prefer more specific type over `'unknown'`; warn on genuine conflict (e.g., same tokenId as both `'color'` and `'number'`)
   - Accumulate `encounteredOn` locations across all encounters
4. Supplemental: collect `appearance.fills[*].tokenRef` and `strokes[*].tokenRef` only if not already represented in `node.variables.bindings` вҖ” variables are the canonical source, paint-level refs are supplemental
5. Classify into counts by `resolvedType`: `'color'` вҶ’ colors, `'number'` вҶ’ numbers, `'string'`/`'boolean'`/`'unknown'` вҶ’ other
6. Emit summary with scope metadata

**Output schema** (in `src/schemas/tokens-used.ts`):

```ts
const EncounteredLocationSchema = z.object({
  nodeId: z.string(),
  nodeName: z.string(),
  field: z.string(),
})

const TokenReferenceSchema = z.object({
  tokenId: z.string(),
  tokenName: z.string().nullable(),
  collectionId: z.string().nullable(),
  resolvedType: z.enum(['color', 'number', 'string', 'boolean', 'unknown']),
  encounteredOn: z.array(EncounteredLocationSchema),
})

const StyleReferenceSchema = z.object({
  type: z.enum(['fill', 'stroke', 'text', 'effect', 'grid']),  // constrained placeholder
  id: z.string(),
  name: z.string().nullable(),
  encounteredOn: z.array(EncounteredLocationSchema),
})

const TokensUsedSummarySchema = z.object({
  scope: z.object({
    fileKey: z.string(),
    rootNodeId: z.string(),
    isFullInventory: z.literal(false),
  }),
  variables: z.array(TokenReferenceSchema),
  styles: z.array(StyleReferenceSchema),  // forward-compatibility placeholder вҖ” empty in DE-004

    colors: z.number(),
    typography: z.number(),

```

**`styles`**: Forward-compatibility placeholder. Empty array in DE-004. Style reference extraction is not in scope. Documented as reserved; not a populated DE-004 concept. The `StyleReferenceSchema.type` enum (`fill`, `stroke`, `text`, `effect`, `grid`) is provisional вҖ” may be widened non-breakingly when style extraction is implemented.

**`counts.numbers`**: Named honestly for what the data tells us. Includes all `resolvedType: 'number'` bindings (spacing, sizing, etc.). Presented as "spacing" only in user-facing docs if appropriate.

## 7. Design Decisions & Trade-offs

- **DEC-021** вҖ” Unified `AnalysisResult<T>` base type for extraction and inference. `ExtractorResult<T>` extends with `omittedFields`. Rationale: one confidence/warning protocol avoids parallel conventions. `omittedFields` is extraction-specific and should not leak into inference signatures.

- **DEC-022** вҖ” Named recursive `InferenceInput` type with `variables` excluded. Rationale: inference reads structure and presentation; variable bindings are an extraction output, not an inference input. Named type (not `Pick<NormalizedNode>`) makes the boundary legible and prevents accidental widening.

- **DEC-023** вҖ” Component extraction distinguishes instance property *values* from component property *definitions*. `propertyValues` populated for instances; empty for components/component-sets. Property definitions deferred вҖ” different shape, different consumer. `propertyReferences` is nested-node вҶ’ component-property bindings (per Figma API), not instance overrides.

- **DEC-024** вҖ” Variable binding `modeId` stays node-level only (in `explicitModes`), not per-binding. Without reliable collection resolution per variable, per-binding `modeId` is false precision. Consumers cross-reference when they have Variables API context.

- **DEC-025** вҖ” Asset extraction: image fills are high confidence; vector/boolean-op complexity is heuristic only (medium confidence, "likely export candidate"). Name patterns reinforce but never trigger alone.

- **DEC-026** вҖ” Role inference via priority-ordered first-match rule chain. 13 rules, explicit priority resolves ambiguity. `null` result = "no match" (distinct from any semantic label). Rules 5вҖ“7 guarded by `type === 'text'`. Label uses "same parent as input-like sibling" вҖ” positional, not spatial.

- **DEC-027** вҖ” Text kind derived from role where possible, independent fallback otherwise. Eliminates duplicate heuristics in the common case. Optional `parentRole` parameter covers nested text (text inside button-role frame).

- **DEC-028** вҖ” Semantic flags are derivations from role/type/component, not a competing classifier. Cheap consequences of prior decisions. Confidence inherits from role.

- **DEC-029** вҖ” Confidence bands: high = strong match, no competing signals; medium = partial match or extractor degradation; low = conflicting signals or failed discrimination. `unknown` is not automatically low. Noise nodes: high-confidence null (clearly ignorable) vs low-confidence null (ambiguous unmatched).

- **DEC-030** вҖ” Tokens-used summary is a post-normalization tree walk, not an in-pipeline accumulator. Lives in `src/summary/`, not `normalize/`. Design rule: anything that aggregates across nodes belongs to summary/orchestrate, not extraction. Variables are canonical source; paint-level tokenRef is supplemental. Dedup prefers more specific resolvedType on conflict.

- **DEC-031** вҖ” Inference is a top-down post-pass over the completed extraction tree, not interleaved with assembly. `normalizeNode` builds the full tree with defaults; `applyInferencesRecursive` walks top-down, making parent role available for child text-kind inference. Label rule reads sibling structural signals directly (name, bounds, appearance), not sibling inferred roles, avoiding inference ordering dependencies.
