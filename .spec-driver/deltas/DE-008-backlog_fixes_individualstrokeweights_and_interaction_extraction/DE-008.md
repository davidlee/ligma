## 1. Summary & Context
- **Product Spec**: PROD-001 – Figma fetch and normalization pipeline
- **Technical Spec**: SPEC-001 – Figma fetch and normalization system
- **Implementation Plan**: [IP-008](./IP-008.md)
- **Change Drivers**: ISSUE-001 (p2, data fidelity), ISSUE-002 (p3, missing extraction)

## 2. Motivation
- `extractAppearance()` reads scalar `strokeWeight` but ignores `individualStrokeWeights`, producing incorrect CSS for per-side strokes (e.g. bottom-only underlines rendered as uniform `border: 1px`).
- Figma `interactions` array (trigger/action pairs for hover, click, prototyping transitions) is dropped entirely during normalization — losing useful implementation context for interactive components.
- Both are gaps in the normalization engine's fidelity (SPEC-001 capability: normalization-engine).

## 3. Scope & Objectives
- **Primary Outcomes**:
  - Per-side stroke weights extracted and represented in `NormalizedStroke` (FR-006)
  - Figma interactions extracted into a new `NormalizedInteraction` schema field
- **Operational Constraints**: Neither change affects existing public API shape for other normalized fields; additive schema extensions only.
- **Dependencies**: None — builds on existing DE-003/DE-004 normalization work.

## 4. Out of Scope
- Full interaction simulation or prototyping replay
- Variable/conditional/media-runtime action families (normalize to `unknown` with warnings; future delta can extend)
- Interaction transition metadata (duration, easing, overlay position)
- Variable-driven interaction bindings

## 5. Approach Overview
- **System Touchpoints**: `src/normalize/appearance.ts`, `src/schemas/normalized.ts`, new or extended extractor for interactions
- **Key Changes**:
  - Extend `NormalizedStroke` to support per-side weights when `individualStrokeWeights` present (FR-006)
  - Prefer `individualStrokeWeights` over scalar `strokeWeight` when both exist
  - Add `NormalizedInteraction` type and extract `interactions` array from raw nodes
  - Extend `NormalizedNode` with optional `interactions` field
- **Migration / Rollout Notes**: Additive — no breaking changes to existing output consumers.

## 6. Verification Strategy
- **Requirements Coverage**: VT-009 (appearance extraction) extended for per-side strokes; new VT for interaction extraction
- **Acceptance Criteria**:
  - Node with `individualStrokeWeights: { top: 0, right: 0, bottom: 2, left: 0 }` produces per-side output, not `weight: 1`
  - Node with `interactions: [{ trigger: ON_HOVER, action: CHANGE_TO }]` produces normalized interaction data
  - Existing appearance tests remain green
  - `mise run` passes

## 7. Risks & Mitigations
- **Risk**: Schema extension breaks downstream consumers – *Likelihood*: low – *Impact*: medium – *Mitigation*: Additive-only changes; new fields are optional.
- **Risk**: Edge cases in Figma interaction data shape – *Likelihood*: medium – *Impact*: low – *Mitigation*: Conservative extraction; unknown trigger/action types logged as warnings.

## 8. Follow-ups & Tracking
- **Backlog Items**: ISSUE-001 (resolves), ISSUE-002 (resolves)
- **Open Decisions**: Which interaction trigger/action types to support in v1 — scoped to ON_HOVER, ON_CLICK initially.
