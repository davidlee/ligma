# DR-002 – Project scaffold and Figma client

## 1. Executive Summary
- **Delta**: [DE-002](./DE-002.md)
- **Status**: draft
- **Last Updated**: 2026-03-10
- **Synopsis**: Greenfield TypeScript project scaffold with Figma API client, URL parser, image export, error hierarchy, and artifact output pipeline.

## 2. Problem & Constraints
- **Current Behaviour**: No code exists. PROD-001 and SPEC-001 define the target system; this delta creates the foundation.
- **Drivers / Inputs**: SPEC-001 FR-001 through FR-004, FR-015, FR-016, NF-004
- **Constraints / Guardrails**: Minimal dependencies (NF-006). Library-first architecture (NF-005). Node 18+ for native fetch.
- **Out of Scope**: Normalization, outline generation, context.md, caching, selective expansion. Asset download (`fetch-assets.ts`) — deferred to DE-004.

## 3. Architecture Intent

### Target Outcomes
- Functional CLI: `figma-fetch "<url>" --token "$TOKEN" --out ./artifacts` (FR-001, NF-005)
- URL parser returning `{ fileKey, nodeId, originalUrl }` (SPEC-001.FR-001)
- Swappable auth module: `{ token, header() }` contract (SPEC-001.FR-002)
- HTTP client with retry + rate-limit backoff (SPEC-001.FR-003)
- Node subtree fetch via `/v1/files/:key/nodes` (SPEC-001.FR-003)
- Image export via GET `/v1/images/:key` — two-step: render request (returns presigned URLs) → download binary from URL (SPEC-001.FR-004)
- Artifact directory writer + manifest.json (SPEC-001.FR-015)
- Typed error hierarchy (SPEC-001.FR-016)

### Client Design Stance
- `FigmaClient` is a stateless, injectable service — not a one-shot pipeline step
- Consumers (normalizer in DE-003, expansion logic in DE-006) receive the client instance and call `fetchNode` repeatedly
- All outbound Figma API calls are wrapped with `p-limit` inside the client

### Guiding Principles
- Every module exports pure functions or thin classes with no hidden state
- Errors carry context (node ID, file key, HTTP status) — not just messages
- Raw API response types are Zod-validated at the boundary; internal code uses validated types
- CLI orchestrates; it does not contain business logic

**Orchestration** (`src/orchestrate.ts`, DEC-006):
```ts
/**
 * Pure data result from orchestration — no I/O.
 * rawNode is `unknown` intentionally: DE-002 passes through raw API JSON.

type OrchestrateResult = {
```

**CLI Error Presentation**:
On failure, CLI prints `error.message` to stderr. If `error.context` is present, prints it on subsequent lines as pretty-printed JSON (`JSON.stringify(context, null, 2)`). Exits with code 1. No `--verbose` flag in DE-002 scope.

## 7. Design Decisions & Trade-offs
- **DEC-001**: commander over yargs. Rationale: lighter, sufficient for the simple flag set. yargs' auto-help is nice but not worth the extra weight for 6 flags.
- **DEC-002**: Native fetch (Node 18+) over undici. Rationale: one fewer dependency. undici can be swapped in later if needed (same API surface).
- **DEC-003**: Zod `.passthrough()` on raw API schemas. Rationale: Figma may add fields; we should validate what we use without rejecting unknown properties.
- **DEC-004**: Error hierarchy uses `class extends` with `cause` chaining. Rationale: standard JS error pattern, instanceof checking, cause chain for debugging.
- **DEC-005**: `--out` is the bundle root — one invocation, one directory, one manifest. Caller owns directory naming. Rationale: no magic, predictable output location. Future wrapper anticipated to generate unique subdirs (e.g. `nanoid(7)`) and return the path on stdout.
- **DEC-006**: Separate `orchestrate()` (pure data assembly) from `writeOutput()` (I/O). Rationale: library-first (NF-005) — MCP wrapper and other consumers compose `orchestrate → their own I/O` without being forced through filesystem writes. CLI does `orchestrate → writeOutput`. Near-zero cost vs inlining.
