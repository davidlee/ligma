## 1. Summary & Context
- **Product Spec**: PROD-001 – Figma fetch and normalization pipeline
- **Technical Spec**: SPEC-001 – selective-expansion capability + caching (NF-005)
- **Implementation Plan**: [IP-006](./IP-006.md)
- **Change Drivers**: DE-001 planning delta

## 2. Motivation
Large or deeply nested frames need staged retrieval. Initial depth-2 fetch may leave important children unresolved. This delta adds trigger-based selective expansion (refetch specific children when normalization detects ambiguity) and file-based caching (reduce API cost and improve determinism during iterative work).

## 3. Scope & Objectives
- **Primary Outcomes**:
  - Expansion trigger evaluation after initial normalization (FR-015)
  - Selective child node refetching via Figma API (FR-015)
  - Merge expanded nodes into normalized tree (FR-015)
  - File-based cache under `.cache/figma-fetch` (FR-014)
  - Cache key: file key + node ID + depth + version + fetch flags (FR-014)
- **Dependencies**: DE-002 (Figma client for refetching), DE-003 (normalization for trigger evaluation)

## 4. Out of Scope
- Core normalization (DE-003)
- Semantic inference (DE-004)
- Outline/context generation (DE-005)

## 5. Approach Overview
- **System Touchpoints**: `src/figma/fetch-node.ts` (expansion refetch), `src/normalize/index.ts` (trigger eval + merge), `src/util/fs.ts` + `src/util/hash.ts` (cache)
- **Key Changes**:
  - Expansion triggers in normalizer: many descendants, incomplete layout containers, text needing deeper inspection, component instances needing metadata, vector/icon nodes, image fills, incomplete variable context
  - Refetch loop: normalizer → trigger list → client fetches → normalizer merges
  - Cache layer wrapping Figma client calls
  - Cache invalidation by version/flags

## 6. Verification Strategy
- Unit tests for each expansion trigger
- Integration test: shallow fetch → trigger → refetch → merged tree
- Unit tests for cache key generation
- Integration test: cached vs uncached fetch produces identical output
- Test cache invalidation on flag/version change

## 7. Risks & Mitigations
- **Risk**: Expansion loop fetches too many children – *Likelihood*: medium – *Impact*: medium – *Mitigation*: Configurable max expansion depth/count, p-limit concurrency
- **Risk**: Cache staleness – *Likelihood*: low – *Impact*: low – *Mitigation*: Version-aware cache keys, manual cache clear option
