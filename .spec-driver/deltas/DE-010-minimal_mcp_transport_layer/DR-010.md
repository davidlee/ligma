# DR-010 – Minimal MCP transport layer

## 1. Executive Summary
- **Delta**: [DE-010](./DE-010.md)
- **Status**: draft
- **Last Updated**: 2026-03-12
- **Synopsis**: Add a thin MCP server exposing the ligma pipeline as 6 granular tools over stdio, plus 2 new CLI subcommands for asset discovery/fetch.

## 2. Problem & Constraints
- **Current Behaviour**: The pipeline is only accessible via CLI. Agents using Claude Desktop or other MCP clients cannot invoke ligma directly.
- **Drivers**: Brief Phase 5 ("optional thin MCP wrapper"); PROD-001.NF-005 (library must be importable for MCP).
- **Constraints**:
  - Zero business logic in the MCP layer — transport only
  - Token via `FIGMA_TOKEN` env var, not per-call
  - stdio transport only (no HTTP/SSE)
- **Out of Scope**: Streaming/incremental results, MCP resources, MCP-specific caching or state

## 3. Architecture Intent

### Target structure

```
                    ┌── cli.ts ──────────────────────┐
                    │  default: orchestrate pipeline  │
                    │  list-assets: discovery         │
                    │  get-asset: targeted fetch      │
                    └───────────────┬────────────────┘
                                    │
src/session.ts ─────────────────────┤
  createSession(config)             │
  → { client, cache, parsed }       │
                                    │
                    ┌── mcp.ts ─────┴────────────────┐
                    │  stdio server, 6 tools          │
                    │  each uses session + primitives  │
                    └────────────────────────────────┘
```

### Session interface

```ts
interface Session {
  readonly client: FigmaClient
  readonly cache: Cache
  readonly parsed: { fileKey: string; nodeId: string }
  readonly config: FetchConfig
}

function createSession(config: FetchConfig): Session
```

Extracted from `orchestrate.ts` lines 61–64. Both CLI and MCP use `createSession` as the shared foundation.

### orchestrate refactor

`orchestrate(config)` becomes `orchestrate(session)`. Internal client/cache creation removed. Existing behavior preserved.

## 4. MCP Tool Specifications

Server name: `ligma`. Transport: stdio. Auth: `FIGMA_TOKEN` env var validated at startup.

### figma_get_node
- **Params**: `url`, `depth?`, `includeHidden?`, expansion opts
- **Pipeline**: `fetchNodeCached` → `normalize` → `generateContextMd` → `aggregateTokensUsed`
- **Returns**: inline normalized JSON, context.md, tokens-used

### figma_get_outline
- **Params**: `url`, `depth?`, `includeHidden?`
- **Pipeline**: `fetchNodeCached` → `normalize` → `buildOutline` → `outlineToXml`
- **Returns**: inline outline JSON + XML

### figma_get_render
- **Params**: `url`, `format?`, `scale?`, `outputDir?`
- **Pipeline**: `fetchImageCached`
- **Returns**: writes image to disk, returns path
- **Note in description**: May be slow — consider backgrounding.

### figma_list_assets
- **Params**: `url`, `depth?`
- **Pipeline**: `fetchNodeCached` → `normalize` → `collectExportTargets`
- **Returns**: inline list of `{ nodeId, name, format, reason }`

### figma_get_asset
- **Params**: `url`, `nodeId`, `format?`, `outputDir?`
- **Pipeline**: `fetchImageCached` for specific node
- **Returns**: writes image to disk, returns path
- **Note in description**: May be slow — consider backgrounding.

### figma_get_assets
- **Params**: `url`, `depth?`, `format?`, `outputDir?`, `maxAssets?`
- **Pipeline**: `fetchNodeCached` → `normalize` → `collectExportTargets` → `fetchAssets`
- **Returns**: writes images to disk, returns paths
- **Note in description**: Fetches up to 20 assets. May be slow — consider backgrounding.

### Return strategy
- Text/JSON artifacts: returned inline in MCP tool response
- Images/assets: written to `outputDir` (default `./artifacts`), path returned
- Rationale: inline images bloat agent context window — the whole point of ligma is token efficiency

## 5. CLI Subcommands

```
ligma <url>                          # existing: full pipeline
ligma list-assets <url>              # new: print detected export targets as JSON
ligma get-asset <url> <node-id>      # new: fetch single asset to disk
```

| Subcommand | Shared flags | Notes |
|------------|-------------|-------|
| `list-assets` | `--token`, `--depth`, `--no-cache`, `--cache-directory` | JSON to stdout |
| `get-asset` | `--token`, `--out`, `--format`, `--no-cache`, `--cache-directory` | writes to `outputDir/assets/`, prints path |

## 8. Design Decisions

- **DEC-010-01**: 6 granular tools (per brief Phase 5) over 1 composite tool. Agents fetch only what they need.
- **DEC-010-02**: Session extraction. Shared client/cache/parsed setup used by both CLI and MCP. Avoids duplication without coupling.
- **DEC-010-03**: Inline return for text/JSON, disk write + path for images. Prevents context window bloat.
- **DEC-010-04**: Token via `FIGMA_TOKEN` env var only. Standard MCP pattern, keeps tool schemas clean, avoids credential leakage in tool call logs.
- **DEC-010-05**: stdio transport only. No HTTP/SSE. Simplest possible transport for agent integration.
- **DEC-010-06**: Asset overwrite on fetch. No staleness check — Figma render API provides no ETags/last-modified. Existing cache layer handles intra-session dedup.

## 9. Operational Notes
- `ligma-mcp` bin entry (`dist/mcp.js`) runs the MCP server. `src/mcp.ts` must include `#!/usr/bin/env node` shebang.
- Server fails fast with a clear error if `FIGMA_TOKEN` is unset at startup.
- Configure in Claude Desktop / MCP client config as: `{ "command": "ligma-mcp", "env": { "FIGMA_TOKEN": "..." } }`
- `outputDir` for image-returning tools is relative to the MCP server's working directory (same as CLI).
- No migration needed — purely additive
