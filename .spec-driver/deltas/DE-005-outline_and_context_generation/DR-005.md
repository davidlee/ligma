# DR-005 – Outline and context generation

## 1. Executive Summary
- **Delta**: [DE-005](./DE-005.md)
- **Status**: draft
- **Last Updated**: 2026-03-11
- **Synopsis**: Add sparse outline generation (JSON + XML) and agent-facing context.md briefing to the pipeline. Pure projection/output — no NormalizedNode schema changes.

## 2. Problem & Constraints
- **Current Behaviour**: Pipeline produces normalized JSON and tokens-used summary but no navigation artifact and no agent-facing briefing. Agents must read full normalized JSON to orient. Neither `normalized-node.json` nor outlines are written to disk. Manifest marks these outputs as optional even though the pipeline always has the data to produce them.
- **Drivers / Inputs**: FR-011 (outline), FR-012 (context.md), NF-002 (determinism). PROD-001 capability definitions: `outline-generation`, `context-generation`. DE-003 design principles (extractor independence, absence semantics).
- **Constraints / Guardrails**:
  - ADR-001: zero lint warnings
  - ADR-002: `mise run` quality gate
  - NF-002: deterministic output — identical inputs produce identical outputs
  - NF-003: pure functions, independently testable
  - DE-005 boundary: projection/output only, no changes to `NormalizedNode` schema or extractors
- **Out of Scope**: Selective expansion (DE-006). Caching (DE-006). New normalization extractors. Role inference changes. Mixed-style text.

## 3. Architecture Intent

### Target Outcomes
- Sparse outline in JSON and XML formats for agent navigation (FR-011)
- Agent briefing `context.md` as primary entrypoint artifact (FR-012)
- Deterministic output for all new artifacts (NF-002)
- All pipeline outputs written to disk with manifest tracking
- `--include-hidden` flag for outline visibility control

### Design Boundary

DE-005 is **projection and output only**. It consumes the `NormalizedNode` tree produced by `normalize()` (DE-003 + DE-004) and transforms it into navigation and briefing artifacts. No changes to the normalization pipeline, extractors, inference, or `NormalizedNode` schema.

```
NormalizedNode (from normalize())
  │
  ├─ buildOutline(node, options) → { outline: OutlineNode, hiddenNodesOmitted: number }
  │   ├─ outline → JSON file
  │   └─ outlineToXml(outline) → XML file
  │
  └─ generateContextMd({ node, manifest, tokensUsed, outline, hiddenNodesOmitted }) → string
      └─ context.md file
```

### Guiding Principles
- Outline is a filtered projection — compact navigation view, not lossless structure
- context.md is an agent briefing note — every line disambiguates, prioritizes, or prevents a mistake
- Normalized JSON remains the source of structural truth; visual reference remains the source of visual truth
- Pure functions throughout; no side effects, no environment dependencies
- Determinism follows from pure functions over stable input

## 5. Detailed Design

### 5.1 OutlineNode Schema — `src/schemas/outline.ts`

```ts
import { z } from 'zod'
import { NormalizedNodeTypeSchema, NormalizedRoleSchema } from './normalized.js'

export interface OutlineNode {
  id: string
  name: string
  type: string          // NormalizedNodeType enum value
  role: string | null   // NormalizedRole enum value or null
  visible: boolean
  bounds: { x: number; y: number; width: number; height: number } | null
  childCount: number    // total structural children (pre-filter)
  children: OutlineNode[]
}

export const OutlineNodeSchema: z.ZodType<OutlineNode> = z.lazy(() =>
  z.object({
    id: z.string(),
    name: z.string(),
    type: NormalizedNodeTypeSchema,
    role: NormalizedRoleSchema.nullable(),
    visible: z.boolean(),
    bounds: z.object({
      x: z.number(), y: z.number(),
      width: z.number(), height: z.number(),
    }).nullable(),
    childCount: z.number().int().nonneg(),
    children: z.array(OutlineNodeSchema),
  }),
)
```

`visible` is always present in the schema. In default mode all nodes are visible (hidden filtered out). In `includeHidden` mode, hidden nodes appear with `visible: false`.

`childCount` is the total structural child count from the source `NormalizedNode`, not the filtered count. The discrepancy between `childCount` and `children.length` signals that hidden nodes were filtered.

### 5.2 Outline Generation — `src/normalize/outline.ts`

```ts
export interface OutlineOptions {
  includeHidden?: boolean  // default: false
}

export interface OutlineResult {
  outline: OutlineNode
  hiddenNodesOmitted: number
}

export function buildOutline(
  node: NormalizedNode,
  options?: OutlineOptions,
): OutlineResult

export function outlineToXml(outline: OutlineNode): string
```

**`buildOutline`**: Recursive projection from `NormalizedNode → OutlineNode`.

- Root node is always included regardless of visibility (it is the entry point)
- Filters `!node.visible` children when `includeHidden` is false (default)
- Projects: `id`, `name`, `type`, `role`, `bounds`, `visible`
- `childCount` = `node.hierarchy.childCount` (total, pre-filter)
- Counts hidden nodes omitted across entire tree (returned as `hiddenNodesOmitted`)
- Recurses into children with same options

**`outlineToXml`**: Recursive XML serialization.

- Element name from `NormalizedNodeType` via a const `Record<NormalizedNodeType, string>` mapping — never arbitrary strings
- Attributes in stable order: `id`, `name`, `role` (omit if null), `w`, `h` (from bounds, omit if null), `child-count`
- `visible="false"` attribute added only when node is hidden (only possible in `includeHidden` mode)
- Leaf nodes (empty `children`): self-closing
- Container nodes: nested with 2-space indentation
- Attribute values XML-escaped (`&amp;`, `&quot;`, `&lt;`, `&gt;`) — `name` is user-supplied Figma layer name

Example output:
```xml
<frame id="123:456" name="Login Card" role="card" w="320" h="240" child-count="8">
  <text id="123:457" name="Title" role="heading" child-count="0" />
  <frame id="123:458" name="Form Fields" role="stack" w="280" h="140" child-count="2">
    <instance id="123:459" name="Email" role="input" w="280" h="48" child-count="0" />
    <instance id="123:460" name="Password" role="input" w="280" h="48" child-count="0" />
  </frame>
  <instance id="123:461" name="Primary Button" role="button" w="280" h="44" child-count="0" />
</frame>
```

### 5.3 context.md Generation — `src/output/context-md.ts`

#### Design intent

context.md is an **agent briefing note**, not documentation. Every line should disambiguate implementation, prioritize work, or prevent a likely mistake.

```ts
export interface ContextMdInput {
  node: NormalizedNode
  manifest: Manifest
  tokensUsed: TokensUsedSummary
  outline: OutlineNode
  hiddenNodesOmitted: number
}

export function generateContextMd(input: ContextMdInput): string
```

#### Sections

Each section is a pure function `ContextMdInput → string | null`. Null sections are omitted. Fixed order is preserved.

**1. Source**
```md
## Source
- File key: abc123
- Node ID: 42:1067
- File name: Mobile App v2
- Version: 3847291
```
Omit lines where value is undefined.

**2. Visual reference**
```md
## Visual reference
Use ./visual/42:1067.png as the visual source of truth.
```
Directive framing. Omit section if no image produced.

**3. Structural summary**
```md
## Structural summary
- Root: frame, role card
- Size: 320×240
- Layout: vertical auto-layout, gap 16, padding 24/24/24/24
- Children: 8 total, 3 shown in outline, 5 hidden
```
Compact. Layout as single dense line. Children line shows total + outline + hidden breakdown. Omit null fields.

**4. Important children** (DEC-022)
```md
## Important children
1. Title (heading)
2. Email Field (input)
3. Primary Button (button)
4. Form Fields (stack)
5. Hero Image (image)
… and 3 more children
```

Selection rule:
1. Direct visible children of root
2. Sort by role importance tier, then document order as stable tie-breaker:
   - **Tier 1** (primary interaction): `heading`, `input`, `button`, `icon-button`
   - **Tier 2** (key content): `image`, `icon`, `card`, `badge`, `avatar`
   - **Tier 3** (structural): `stack`, `grid`, `list`, `list-item`, `navigation`, `modal`, `container`, `divider`
   - **Tier 4**: null role (document order)
3. Cap at 8
4. "… and N more children" when truncated
5. Label: `{name} ({role ?? type})`

No secondary heuristic tie-breakers beyond document order.

**5. Tokens used**
```md
## Tokens used
- color.bg.surface
- color.text.primary
- spacing.24
- radius.md
```
Or when names are unresolved:
```md
## Tokens used
12 token references detected in v1 (mostly color/number; names unresolved)
```
Include type skew from `resolvedType` when available. Cap at 15, "… and N more" truncation. Ordered by encounter order (deterministic tree walk).

**6. Assets**
```md
## Assets
- logo (svg, suggested export)
- hero illustration (bitmap, suggested export)
```
No invented filenames. Label: `{node.name} ({asset.kind}, suggested export)`. Only nodes with `asset.exportSuggested`. Cap at 10.

**7. Implementation notes** (DEC-023)

Six generators, each `ContextMdInput → string | null`:

| Generator | Fires when | Example output |
| --- | --- | --- |
| `layoutNote` | Root has auto-layout | "Prefer vertical stack layout; root uses auto-layout with 16px gap" |
| `componentNote` | Root has component info | "Root is a component instance (Button/Primary); treat as reusable" |
| `complexityNote` | Deep nesting (>3 levels) or many children (>12) or many token refs (>8) | "Structure is moderately deep (4 levels); implement incrementally" |
| `assetNote` | Descendants have `exportSuggested` | "3 nodes suggested for asset export; prefer exported assets over recreating vectors" |
| `interactionNote` | Descendants have interactions | "Contains 2 interactive elements; ensure event handlers are wired" |
| `truncationNote` | `hiddenNodesOmitted > 0` | "Outline omits 5 hidden nodes; normalized JSON retains the full tree" |

Collect non-null, sorted by: layout → component → complexity → asset → interaction → truncation. Cap at 5.

```md
## Implementation notes
- Prefer vertical stack layout; root uses auto-layout with 16px gap
- Root is a component instance (Button/Primary); treat as reusable
- Structure is moderately deep (4 levels); implement incrementally
```

#### Language stance

- Directive where data supports it ("prefer", "treat as", "ensure")
- Factual otherwise — no interpretive certainty beyond what data warrants
- No invented filenames
- No "must/always" language when only heuristics support the claim
- Do not restate information already obvious from section headings unless it changes action

#### Invariants

- Deterministic: no timestamps, no random IDs, no environment data
- Flat markdown, no nested structure beyond bullet lists
- Fixed section order, empty sections omitted
- Visual reference = source of visual truth
- Normalized JSON = source of structural truth

## 8. Design Decisions & Trade-offs

- **DEC-021**: Outline omits hidden nodes by default. `includeHidden` option restores them with `visible="false"`. context.md notes omission count. Normalized JSON remains the source of full structural truth. Rationale: outline is for implementation navigation, not forensic completeness. Simple visibility filter over opinionated heuristics (hidden leaves vs containers).
- **DEC-022**: Important children sorted by role importance tier (heading/input/button > image/icon/card > stack/grid/container > null role), then document order as stable tie-breaker. Cap at 8. No secondary "clever" tie-breakers. Compact label: `{name} ({role ?? type})`. Rationale: surfaces implementation-relevant children first without overcomplicating the sort.
- **DEC-023**: Implementation notes: fixed rule set of 6 pure `NormalizedNode → string | null` generators. Root-level facts only unless child-level fact is unusually salient. Cap at 5. Directive language tied to evidence ("prefer", "treat as"), not interpretive certainty. Do not restate information obvious from section headings. Rationale: useful hints, bounded scope, independently testable, easy to prune.
- **DEC-024**: Manifest outputs: `normalizedNodeJson`, `outlineJson`, `outlineXml`, `contextMd`, `tokensUsedJson` become required. `png`, `svg`, `assets` remain optional. Rationale: pipeline always produces these; optional typing creates unnecessary defensive code.
- **DEC-025**: `--include-hidden` wired end-to-end: config → orchestrate → buildOutline → CLI flag. Default `false`. Rationale: directly tied to DE-005's primary outputs, bounded risk, avoids gap between designed behavior and user reachability.
