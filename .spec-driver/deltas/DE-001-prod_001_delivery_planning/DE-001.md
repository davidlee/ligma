## 1. Summary & Context
- **Product Spec**: PROD-001 – Figma fetch and normalization pipeline
- **Technical Spec**: To be created (SPEC-001, assembly) as output of this delta
- **Implementation Plan**: [IP-001](./IP-001.md)
- **Change Drivers**: Initial project setup from `docs/brief.md`

## 2. Motivation
PROD-001 defines a substantial system with ~15 functional requirements spanning URL parsing, API integration, a normalization engine, outline generation, context generation, caching, and selective expansion. Before implementation begins, the work needs to be decomposed into a TECH assembly spec (defining the system's module architecture and responsibilities) and a series of scoped delivery deltas, each small enough to implement and verify independently.

## 3. Scope & Objectives
- **Primary Outcomes**:
  - TECH assembly spec (SPEC-001) capturing system architecture, module boundaries, and requirement allocation
  - Ordered series of scoped delivery deltas, each targeting a coherent slice of PROD-001 requirements
  - Each delivery delta scoped with clear applies-to requirements and verification expectations
- **Operational Constraints**: Planning only — no implementation code in this delta
- **Dependencies**: None (first delta in the project)

## 4. Out of Scope
- Implementation code
- MCP wrapper (deferred per PROD-001)
- OAuth support (deferred per PROD-001)

## 5. Approach Overview
- **Phase 1**: Draft TECH assembly spec from PROD-001 requirements and brief architecture
- **Phase 2**: Decompose PROD-001 into delivery deltas following the brief's phased implementation strategy, adapted for spec-driver governance:
  1. Project scaffold + URL parsing + auth + client + raw fetch + image export + file output + manifest
  2. Normalization engine: layout, text, appearance, bounds, hierarchy
  3. Role inference + component/instance + token/variable extraction + asset classification + diagnostics
  4. Outline generation + context.md generation
  5. Selective expansion + caching
- **Phase 3**: Review delta dependency graph, validate requirement coverage

## 6. Verification Strategy
- **Requirements Coverage**: Each PROD-001 FR/NF must appear in exactly one delivery delta's applies-to
- **Acceptance Criteria**: All PROD-001 requirements allocated to delivery deltas; TECH spec created and consistent with PROD-001

## 7. Risks & Mitigations
- **Risk**: Delta granularity too fine → overhead – *Likelihood*: low – *Impact*: medium – *Mitigation*: Target 4-6 deltas, not one per requirement
- **Risk**: TECH spec becomes stale during implementation – *Likelihood*: medium – *Impact*: low – *Mitigation*: Revision-driven updates as implementation reveals reality
