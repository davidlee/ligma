# DR-001 – PROD-001 delivery planning

## 1. Executive Summary
- **Delta**: [DE-001](./DE-001.md)
- **Status**: draft
- **Last Updated**: 2026-03-10
- **Synopsis**: Decompose PROD-001 into a TECH assembly spec and a series of scoped delivery deltas for incremental implementation.

## 2. Problem & Constraints
- **Current Behaviour**: PROD-001 exists as a monolithic product spec with 15 FRs and 6 NFRs. No technical architecture or delivery plan exists.
- **Drivers / Inputs**: `docs/brief.md` provides implementation phasing guidance and a recommended project structure.
- **Constraints / Guardrails**: Each delta must be independently implementable and verifiable. Dependency ordering must allow incremental delivery.
- **Out of Scope**: Implementation code, MCP wrapper, OAuth.

## 3. Architecture Intent

### TECH Assembly Spec (SPEC-001)

The assembly spec will define system-level architecture with these module boundaries (derived from brief §Project structure):

| Module | Responsibility | Key PROD-001 Requirements |
| --- | --- | --- |
| `figma/` | URL parsing, auth, API client, node/image/asset fetching | FR-001, FR-002, FR-003, FR-004 |
| `normalize/` | Classification, layout, text, appearance, component, variable, asset normalization; role inference; heuristics; outline generation | FR-005–FR-011 |
| `output/` | Manifest, context.md, file writing | FR-012, FR-013 |
| `schemas/` | Zod schemas for raw, normalized, outline, manifest types | NF-005 |
| `cli` | CLI entry point, config | NF-005, NF-006 |
| `util/` | Caching, retry, logging, hashing | FR-014, NF-004 |

Cross-cutting: FR-015 (selective expansion) spans `figma/` and `normalize/`.

### Delivery Delta Decomposition

| Delta | Title | Requirements | Dependencies |
| --- | --- | --- | --- |
| DE-002 | Project scaffold + Figma client | FR-001, FR-002, FR-003, FR-004, FR-013, NF-004, NF-005, NF-006 | None |
| DE-003 | Core normalization engine | FR-005, FR-006, FR-008, FR-009, NF-001, NF-003 | DE-002 |
| DE-004 | Semantic inference + tokens + assets | FR-007, FR-010, component/variable/asset parts of FR-005 | DE-003 |
| DE-005 | Outline + context.md generation | FR-011, FR-012, NF-002 | DE-003 |
| DE-006 | Selective expansion + caching | FR-014, FR-015 | DE-003 |

### Guiding Principles
- Each delta produces working, tested code
- Normalization deltas (DE-003, DE-004) are the highest-value work
- DE-005 and DE-006 can be parallelized after DE-003
- Fixture-driven testing from the start

## 7. Design Decisions & Trade-offs
- **DEC-001**: 5-delta decomposition. Rationale: matches brief's 5-phase strategy, each delta is independently verifiable, normalization gets 2 deltas reflecting its complexity. Alternative considered: finer per-module deltas (rejected — too much overhead for a small project).
- **DEC-002**: Assembly spec covers module boundaries and requirement allocation but not internal module design. Rationale: unit specs can be created as needed during implementation deltas; assembly spec provides the architectural frame.
