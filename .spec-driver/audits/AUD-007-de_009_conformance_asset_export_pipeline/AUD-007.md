## Observations

### F-001: Asset collection wired correctly
- `orchestrate.ts` calls `collectAndFetchAssets()` after expansion, before manifest build
- Targets collected via `collectExportTargets(normalizedNode, config.maxAssets)`
- Asset paths computed via `assetFileName()` and prefixed with `assets/`
- Paths passed to `buildManifest` in `outputs.assets`
- **disposition: aligned**

### F-002: Fetcher adapter matches DR-009 DEC-004
- `ImageFetcher` adapter wraps `fetchImageCached` — reuses same cache/concurrency as expansion
- `(nodeId, format) => fetchImageCached(client, cache, fileKey, nodeId, { format, version: null }).then(r => r.result.buffer)`
- Matches the design doc's specified adapter shape exactly
- **disposition: aligned**

### F-003: Per-asset failure non-blocking (FR-004 contract)
- `fetchAssets()` catches per-asset errors, records as `ManifestError` with type `AssetExportError`
- Errors merged into manifest errors array in orchestrate
- Pipeline continues on individual asset failure
- **disposition: aligned**

### F-004: Asset writing in output/write.ts (FR-015)
- `writeAssets()` iterates `FetchedAsset[]`, uses `assetFileName()` for filenames, writes to `assets/` subdir
- Called from `writeOutput()` only when assets is defined and non-empty
- `OutputArtifacts.assets` is optional (`readonly FetchedAsset[] | undefined`) — backward compatible
- **disposition: aligned**

### F-005: Format override maps correctly (DR-009 DEC-001, DEC-002)
- `resolveAssetFormatOverride()` maps `'auto'` → `null` (kind-driven), `'png'`/`'svg'` → direct override
- `formatsForKind()` in fetch.ts handles bitmap→PNG, svg→SVG, mixed→both
- **disposition: aligned**

### F-006: FR-015 spec text does not mention asset export
- FR-015 says: "MUST create the artifact directory structure ... and write all artifacts"
- The `assets/` subdir was already in the directory list, and `manifest.outputs.assets` was already in the schema
- But the spec doesn't explicitly describe asset export behavior (collection, fetching, format logic, per-asset error handling)
- This is new capability that should be traceable to a spec requirement
- **disposition: spec_patch** — FR-015 should mention asset writing; FR-011 should mention the collect→fetch pipeline as post-normalization behavior

### F-007: OrchestrateResult.assets is required (not optional)
- Phase sheet said `assets?: readonly FetchedAsset[] | undefined` for OrchestrateResult
- Implementation uses `readonly assets: readonly FetchedAsset[]` (required, empty array when none)
- This is cleaner — callers don't need to null-check. Write side uses optional for backward compat.
- No drift; this is a sound adaptation.
- **disposition: aligned**

## Recommendations

### Spec patch (F-006)
FR-015 and FR-011 in SPEC-001 should be patched to document:
- FR-011: asset collection from normalized tree feeds into fetch pipeline (not just classification)
- FR-015: `writeAssets()` writes fetched assets to `assets/` with `{sanitized-name}-{node-id}.{ext}` filenames; `manifest.outputs.assets` lists relative paths

This is a minor spec patch — the capability exists and is tested; the spec text just needs to catch up.
