## 3. Findings

### F-001: PROD-001 coverage block missing FR-014 and FR-015

- **Severity**: blocking
- **Outcome**: PROD-001 `supekku:verification.coverage@v1` has no entries for FR-014 (caching) or FR-015 (selective expansion). Coverage gate will fail on `complete delta`.
- **Disposition**: `spec_patch`
- **Action**: Add verified entries for FR-014 and FR-015 referencing DE-006 test evidence.

### F-002: SPEC-001 coverage block missing FR-009 and FR-017

- **Severity**: blocking
- **Outcome**: SPEC-001 `supekku:verification.coverage@v1` has no entries for FR-009 (caching) or FR-017 (selective expansion). These are the tech-spec equivalents of PROD-001.FR-014/FR-015.
- **Disposition**: `spec_patch`
- **Action**: Add verified entries for FR-009 and FR-017 referencing DE-006 test evidence.

### F-003: SPEC-001 module architecture stale — cache path and missing modules

- **Severity**: non-blocking
- **Outcome**: SPEC-001 §3 module architecture references `util/cache.ts` for FR-009. Implementation created `src/cache/` (types, keys, store, index) as a dedicated module. `src/expand/` (types, triggers, merge) is also absent from the architecture listing.
- **Disposition**: `spec_patch`
- **Action**: Update module architecture to reflect `src/cache/` and `src/expand/` modules. Update architecture diagram.

### F-004: SPEC-001 FR-017 lists 7 trigger categories; only 2 implemented

- **Severity**: non-blocking
- **Outcome**: SPEC-001.FR-017 and PROD-001.FR-015 list 7 expansion trigger categories as MUST requirements. DR-006 explicitly scoped v1 to 2 triggers (depthTruncatedContainer, geometryNeeded) with extensibility for future additions. The unimplemented triggers are:
  - text nodes needing deeper inspection
  - image fills needing extraction
  - incomplete variable binding context
  - child count exceeding configurable threshold (subsumed by depthTruncatedContainer)
  - component instances needing referenced metadata (partially covered via instance inclusion in container types)
- **Disposition**: `spec_patch`
- **Action**: Patch FR-017 (SPEC-001) and FR-015 (PROD-001) to reflect v1 scope: 2 concrete triggers with extensible architecture for future additions. Move unimplemented triggers to backlog hooks.
- **drift_refs**: DR-006 DEC-036 (single expansion pass only, v1)

### F-005: IP-006 verification coverage entries still `planned`

- **Severity**: non-blocking
- **Outcome**: All 16 VT entries in IP-006's `supekku:verification.coverage@v1` block show `status: planned` despite all tests passing. Should be `verified`.
- **Disposition**: `spec_patch`
- **Action**: Update all 16 entries to `status: verified`.

### F-006: VT ID collision between IP-006 and SPEC-001

- **Severity**: non-blocking
- **Outcome**: IP-006 allocates VT-023 through VT-038 for DE-006 tests. SPEC-001 already uses VT-023 through VT-027 for earlier delta work (DE-004, DE-005, DE-008) referencing different tests. The IDs refer to different test artifacts in different scopes.
- **Disposition**: `follow_up_backlog`
- **Action**: Future housekeeping — renumber DE-006 plan-level VTs or establish VT ID allocation convention. Not blocking because the actual test files are unambiguous and spec-level coverage uses distinct entries.

### F-007: pluginData in NodeCacheKey hardcoded to `'none'`

- **Severity**: non-blocking
- **Outcome**: `fetchNodeCached` hardcodes `pluginData: 'none'` in the cache key. When pluginData endpoint options are wired through, the cache key must include the actual pluginData value to avoid stale hits.
- **Disposition**: `follow_up_backlog`
- **Action**: Create backlog item. Acceptable for v1 since pluginData option is not yet exposed to users.

### F-008: fetchImageCached sourceUrl empty string on cache hit

- **Severity**: non-blocking
- **Outcome**: On cache hit, `fetchImageCached` returns `sourceUrl: ''` because the original source URL is stored only in the metadata sidecar and not read back. sourceUrl is a diagnostic field, not a correctness field.
- **Disposition**: `tolerated_drift`
- **Rationale**: Cosmetic only. sourceUrl is used for logging/debugging, not for downstream processing. The metadata sidecar preserves the original URL for manual inspection.

### F-009: fetchExpansionTargets sequential, not concurrent

- **Severity**: non-blocking
- **Outcome**: Expansion target fetches run sequentially. DR-006 §10 mentions "concurrent via existing client p-limit" but the implementation uses sequential iteration. For typical maxTargets (10), the performance difference is small.
- **Disposition**: `follow_up_backlog`
- **Action**: Future performance delta. The existing client already has p-limit; wiring concurrency into fetchExpansionTargets is straightforward.

### F-010: variant-set in container trigger types

- **Severity**: non-blocking
- **Outcome**: `depthTruncatedContainer` includes `variant-set` in its container type list per DR-006 §6. If real-world data shows excessive false positives, the trigger type list should be narrowed.
- **Disposition**: `tolerated_drift`
- **Rationale**: DR-006 explicitly includes variant-set. False positives are harmless (refetch confirms no children). Monitor in production use.
